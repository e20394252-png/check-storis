# 📋 Спецификация: Бот регистрации на мероприятие за репост

**Проект:** bot-repost (рабочее название)  
**Дата:** Апрель 2026  
**Основа:** Кодовая база проекта `bot-storis` (Биржа Сторис)

---

## 1. Концепция и суть продукта

Система регистрации на мероприятие, где условием участия является репост публикации.

**Механика простая:**
1. Пользователь хочет попасть на мероприятие
2. Он делает репост указанной публикации (сторис/пост)
3. Присылает скриншот репоста через Mini App или бот
4. Админ смотрит скрин и принимает решение: это репост или нет
5. При одобрении — пользователь получает статус «участника» в приложении и уведомление в боте

**Ключевое отличие от Биржи Сторис:**
- Нет коммерческой составляющей — никаких выплат, балансов, вывода средств
- Нет матчинга, кампаний, рекламодателей
- Нет AI/n8n — только человек-админ принимает решение
- Простая бинарная логика: зарегистрирован / не зарегистрирован

## 2. Роли пользователей

### 👤 Участник (User)
- Регистрируется через Telegram Mini App или напрямую через бота
- Делает репост указанной публикации
- Загружает скриншот репоста
- Ожидает проверки
- При одобрении — видит плашку «Участник подтверждён» в приложении и получает уведомление в боте

### 🔑 Администратор (Admin)
- Открывает защищённую веб-страницу `/admin?key=SECRET`
- Видит все заявки со скриншотами
- Нажимает «✓ ТО» или «✗ НЕ ТО» по каждой заявке
- Система автоматически уведомляет пользователя о решении через бота

## 3. Пользовательский путь (User Journey)

### Шаг 1 — Вход через Telegram-бота
```
Пользователь открывает бота → /start
Бот присылает приветственное сообщение + кнопку "📱 Открыть приложение"
(либо бот сразу принимает скрин через диалог — см. альтернативу ниже)
```

### Шаг 2 — Онбординг в Mini App
Mini App открывается внутри Telegram. Пользователь видит:
- Краткое описание мероприятия (что за ивент, дата, где)
- Условие участия: сделать репост публикации (с прямой ссылкой/кнопкой на публикацию)
- Кнопка «Я сделал репост — загрузить скриншот»

### Шаг 3 — Загрузка скриншота
- Пользователь выбирает фото из галереи (нативный File Picker Telegram Mini App)
- Фото загружается на сервер (в базу или внешнее хранилище)
- Статус заявки: **`pending`** (на проверке)
- Пользователь видит сообщение: «Скриншот отправлен! Ожидайте проверки — обычно это занимает до 24 часов.»

### Шаг 4 — Ожидание (состояние в приложении)
Пользователь может открыть Mini App снова и увидеть текущий статус:
- 🕐 `pending` — «Твой репост на проверке»
- ✅ `approved` — «Ты в списке участников! Мероприятие: [дата/место]»
- ❌ `rejected` — «Скриншот не прошёл проверку. Можешь загрузить новый.»

### Шаг 5 — Решение администратора
Админ в панели видит карточку с:
- Telegram username / имя пользователя
- Скриншот репоста (fullscreen)
- Две кнопки: **✓ ТО** и **✗ НЕ ТО**

### Шаг 6 — Уведомление пользователя
После решения бот автоматически:
- **Если одобрено:** «🎉 Поздравляем! Ваш репост подтверждён. Вы в списке участников мероприятия [Название]! До встречи [дата/место].»
- **Если отклонено:** «⚠️ К сожалению, скриншот не прошёл проверку. Убедитесь что репост виден в вашем профиле и загрузите новый скришнот.»

## 5. Схема базы данных (новая, упрощённая)

```prisma
// prisma/schema.prisma

generator client {
  provider        = "prisma-client-js"
  previewFeatures = ["driverAdapters"]
}

datasource db {
  provider = "postgresql"
}

model User {
  id          String   @id @default(uuid())
  telegram_id BigInt   @unique
  username    String?
  first_name  String?
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt

  registration Registration?
}

model Event {
  id          String   @id @default(uuid())
  title       String
  description String?
  date        DateTime?
  location    String?
  repostUrl   String?  // ссылка на публикацию для репоста
  isActive    Boolean  @default(true)
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt

  registrations Registration[]
}

model Registration {
  id          String   @id @default(uuid())
  userId      String   @unique  // один пользователь — одна регистрация (или per event)
  eventId     String
  status      RegistrationStatus @default(PENDING)
  proofUrl    String?  // URL загруженного скриншота
  adminNote   String?  // необязательная заметка от админа при отклонении
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt

  user  User  @relation(fields: [userId], references: [id], onDelete: Cascade)
  event Event @relation(fields: [eventId], references: [id])
}

enum RegistrationStatus {
  PENDING   // скриншот загружен, ожидает проверки
  APPROVED  // одобрен, пользователь — участник
  REJECTED  // отклонён, пользователь может загрузить снова
}
```

**Пояснения по схеме:**
- `User` — минималистичный: только Telegram ID, username, имя
- `Event` — мероприятие. На старте одно активное, в будущем можно несколько
- `Registration` — связка пользователь ↔ мероприятие + скриншот + статус
- `proofUrl` — URL изображения (загружается через API, хранится или в базе как base64/external URL, или на S3/Cloudinary)
- Нет балансов, транзакций, наград — ничего коммерческого

## 6. API эндпоинты

### `POST /api/register`
Подача заявки / загрузка скриншота репоста.

**Вход:**
```json
{
  "initData": "...",     // Telegram Mini App initData
  "eventId": "uuid",
  "proofBase64": "..."   // base64 изображения (или FormData с файлом)
}
```

**Логика:**
1. Валидация `initData` через `validateInitData()` (берём из `lib/twa.ts` без изменений)
2. Парсинг пользователя из `initData`
3. `upsert` на `User` — создать если не существует
4. Загрузить изображение → сохранить `proofUrl`
5. `upsert` на `Registration` — создать с `status: PENDING`, или обновить `proofUrl` у существующей (для повторных попыток после `REJECTED`)
6. Вернуть `{ success: true, status: "pending" }`

**Вниз по стеку:** никаких вызовов n8n, никакой AI, никаких уведомлений при подаче — просто запись в БД.

---

### `GET /api/me`
Получение текущего статуса регистрации пользователя.

**Вход:** заголовок `X-Telegram-Init-Data: ...`

**Выход:**
```json
{
  "user": { "first_name": "Иван", "username": "ivan123" },
  "registration": {
    "status": "pending",   // или "approved" / "rejected" / null
    "eventId": "uuid",
    "createdAt": "2026-04-18T..."
  },
  "event": {
    "title": "...",
    "date": "...",
    "location": "...",
    "repostUrl": "..."
  }
}
```

---

### `POST /api/admin/review` *(Server Action в admin/page.tsx)*
Решение администратора по заявке. Реализуется как Next.js Server Action (аналогично `verifyProof` в `bot-storis`).

**Параметры формы:**
- `registrationId`
- `action: "approve" | "reject"`
- `adminNote` (опционально)
- `_key` (секретный ключ)

**Логика:**
1. Проверка `_key === process.env.ADMIN_SECRET`
2. Найти `Registration` с `include: { user: true, event: true }`
3. Обновить статус: `APPROVED` или `REJECTED`
4. Отправить уведомление пользователю через Telegram Bot API
5. `revalidatePath('/admin')` + `redirect`

---

### `POST /api/webhook/telegram`
Вебхук бота (аналогичен `bot-storis`).

**Обрабатывает:**
- `/start` — приветствие + кнопка открыть Mini App
- `callback_query proof:approve:{registrationId}` и `proof:reject:{registrationId}` — **альтернативный** способ апрува прямо из Telegram (бот присылает сообщение с фото и кнопками)

## 10. Как работает проверка скриншота (детали)

### Загрузка изображения
Telegram Mini App предоставляет нативный `<input type="file">` или через `window.Telegram.WebApp.showScanQrPopup` (не актуально). Используем стандартный `<input type="file" accept="image/*">`.

Изображение конвертируется в base64 на клиенте и передаётся в `POST /api/register`.

### Хранение изображения
Два варианта:

**Вариант A (проще — для MVP):**
Сохранять `proofUrl` как data URL (`base64`) прямо в PostgreSQL в поле `text`. Подходит для небольшого числа участников (до нескольких сотен).

**Вариант B (чище — для масштаба):**
Загружать изображение во внешнее хранилище (Cloudinary free tier / Supabase Storage / любой S3-совместимый) и сохранять URL в БД. Из существующей кодовой базы — уже есть прецедент `base64Image` в `/api/onboarding`, так что механика загрузки готова.

### Отображение в Admin Panel
```tsx
{reg.proofUrl && (
  <img
    src={reg.proofUrl}
    alt="Скриншот репоста"
    style={{ maxWidth: '100%', maxHeight: 500, borderRadius: 12 }}
  />
)}
```
Идентично тому, как отображаются proof-скриншоты в `bot-storis/src/app/admin/page.tsx` (строки 498–502).

## 11. Уведомления — реализация

Используется та же функция `sendMessage` из `lib/notify.ts`, просто с новыми текстами.

```typescript
// lib/notify.ts (новая версия для repost-бота)

export async function notifyRegistrationApproved(
  telegramId: bigint,
  eventTitle: string,
  eventDate?: string,
  eventLocation?: string,
  username?: string | null
) {
  await sendMessage(telegramId,
    `🎉 <b>Регистрация подтверждена!</b>\n\n` +
    `Ты в списке участников <b>${eventTitle}</b> ✅\n\n` +
    (eventDate ? `📅 Дата: ${eventDate}\n` : '') +
    (eventLocation ? `📍 Место: ${eventLocation}\n` : '') +
    `\nДо встречи!`,
    openAppButton
  );
}

export async function notifyRegistrationRejected(
  telegramId: bigint,
  eventTitle: string,
  username?: string | null
) {
  await sendMessage(telegramId,
    `⚠️ <b>Скриншот не прошёл проверку</b>\n\n` +
    `К сожалению, не удалось подтвердить репост для <b>${eventTitle}</b>.\n\n` +
    `Загрузи скриншот повторно — убедись, что репост виден в твоём профиле.`,
    openAppButton
  );
}
```

## 12. Защита Admin Panel

Идентичная механика из `bot-storis`:
- URL: `/admin?key=ВАШ_СЕКРЕТНЫЙ_КЛЮЧ`
- Ключ хранится в `process.env.ADMIN_SECRET`
- Server Component проверяет `searchParams.key !== process.env.ADMIN_SECRET` и показывает заглушку при несовпадении
- Никакой сессии/cookie — просто query param (подходит для внутреннего инструмента)

| **Экспорт списка участников** | Кнопка в админке: скачать CSV со всеми одобренными участниками |
